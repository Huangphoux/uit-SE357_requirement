\section{Introduction}

\subsection{Purpose}

This Software Architecture Document (SAD) describes the software architecture of the \textbf{Learning Management System (LMS)} developed as part of the UIT SE357 course project. It establishes the high-level structure of the system, documents key architectural decisions, and demonstrates how the design addresses the Architecturally Significant Requirements (ASRs) that have been implemented (Status = Done).

The document is intended to serve as the authoritative reference for the current release and as a baseline for future evolution of the system.

\subsection{Scope}

This document covers the architecture of the LMS as deployed via Docker Compose, comprising:

\begin{itemize}[noitemsep]
  \item A \textbf{React SPA} (browser-based front end)
  \item A \textbf{Node.js / Express REST API} (two identical instances for high availability)
  \item A \textbf{PostgreSQL 16} relational database
  \item A \textbf{Redis 7} in-memory data store
  \item A \textbf{Caddy 2} reverse proxy (TLS termination, load balancing, static file serving)
  \item A \textbf{File Storage volume} (uploaded materials and submissions)
\end{itemize}

The document does not cover the business processes the LMS supports; those are described in the BRD and SRS documents.

\section{System Overview}

\subsection{System Description}

The \textbf{Learning Management System (LMS)} is a web-based platform for managing educational courses, classes, learning materials, assignments, and student submissions. It supports three distinct user roles --- Student, Teacher, and Admin --- each with differentiated capabilities.

The system is deployed as a set of Docker containers orchestrated via Docker Compose, exposed to end-users through a Caddy reverse proxy that handles TLS termination and load balancing.

\begin{lstlisting}[caption={System Context Diagram (C4 Level 1)}]
                    +------------------------------+
                    |  Learning Management System  |
                    |       [Software System]      |
                    +------+---------------+-------+
            HTTPS          |               |        SMTP/TLS
    +----------------------+     +--------------------+-----+
    |            |         |     |         |                |
[Student]   [Teacher]  [Admin]  [Email Service]
(person)    (person)   (person)  (external)
\end{lstlisting}

\textbf{System boundaries:}
\begin{itemize}[noitemsep]
  \item Inbound: HTTPS on port 443 (served by Caddy)
  \item Outbound: SMTP to an external email service for notifications
  \item All data stored within the system boundary (PostgreSQL, Redis, file volumes)
\end{itemize}

\textbf{Start-up dependency chain:}
\begin{enumerate}[noitemsep]
  \item \texttt{postgres} becomes healthy (\texttt{pg\_isready} passes)
  \item \texttt{redis} becomes healthy (\texttt{redis-cli ping} passes)
  \item \texttt{backend-primary} and \texttt{backend-secondary} start after (1) and (2)
  \item \texttt{caddy} starts after both backends are healthy; polls \texttt{/api/health} every 10\,s
  \item On \texttt{backend-primary} health failure: Caddy promotes \texttt{backend-secondary} (ASR-AVAIL-01)
\end{enumerate}

\subsubsection{Audit Log Storage}

Audit log entries are stored as \textbf{JSON lines in daily files} under the \texttt{AUDIT\_LOG\_DIR} path, mounted via the \texttt{server-logs} Docker volume. They are \emph{not} stored in PostgreSQL tables. The \texttt{GET /api/audit/logs} endpoint (ADMIN only) reads and returns entries from these files.

Each entry follows the canonical wide-event format:

\begin{lstlisting}[language={}]
{
  "timestamp": "2026-05-04T08:00:00.000Z",
  "userId":    "clv1abc123",

\section{Architecture Decisions}
% ══════════════════════════════════════════════════════════════════

\begin{tabularx}{\textwidth}{L{1.8cm} L{5cm} L{1.8cm} L{5cm}}

\subsection{ADR-001: Caddy as Entry Point}

Caddy 2 serves as the single entry point for all inbound HTTPS traffic. It performs TLS termination, active/passive load balancing between \texttt{backend-primary} and \texttt{backend-secondary}, gzip/zstd compression, static SPA serving, and API health checking every 10 seconds. This single decision simultaneously satisfies ASR-PERF-01 (throughput), ASR-AVAIL-01 (failover), and ASR-SEC-06 (HTTPS).

\subsection{ADR-002: JWT Authentication}

Passwords are hashed with bcrypt ($\geq$ 10 salt rounds). Authentication issues a short-lived JWT access token (1\,h, stored in an \texttt{httpOnly} cookie) and a long-lived refresh token. The stateless access-token design allows both API instances to validate tokens independently without shared session state.

\subsection{ADR-003: Redis Rate Limiting}

Redis \texttt{INCR}/\texttt{EXPIRE} sliding-window counters back the \texttt{rateLimiter} middleware, providing accurate cross-instance rate counting. Login lockout state is persisted in PostgreSQL \texttt{LoginAttempt} for durability. This separates ephemeral rate data (Redis) from durable lockout state (PostgreSQL).

\subsection{ADR-004: PostgreSQL + Prisma}

PostgreSQL 16 is the sole relational store. Prisma ORM enforces parameterized queries throughout, eliminating SQL injection risk (ASR-SEC-09). Explicit \texttt{@@index} directives on \texttt{teacherId}, \texttt{classId}, and \texttt{createdBy} columns support sub-2-second paginated queries on 10\,000+ records (ASR-PERF-02).

\subsection{ADR-005: RBAC}

Three roles (\texttt{STUDENT}, \texttt{TEACHER}, \texttt{ADMIN}) are encoded in the JWT payload. Auth Middleware verifies role claims on every protected route, ensuring 100\,\% RBAC coverage (ASR-SEC-05).

\subsection{ADR-006: Secure File Storage}

Uploaded files are stored in a Docker volume at \texttt{/app/uploads}, outside the Caddy web root. Files are served through the API only (never via direct HTTP). MIME type and file size are validated both client-side and server-side (ASR-SEC-14).

\subsection{ADR-007: Audit Logging}

All security-sensitive actions emit a canonical wide-event log entry. Entries are written as JSON lines to daily files on the \texttt{server-logs} volume. \texttt{AUDIT\_RETENTION\_DAYS=365} governs a scheduled clean-up job satisfying the 1-year retention requirement (ASR-SEC-11).

\subsection{Security}

\subsubsection{ASR-SEC-01 \& ASR-SEC-02 -- Authentication}

Passwords are hashed using \texttt{bcrypt} with a minimum of 10 salt rounds at registration. Authentication returns a JWT access token (1\,h expiry) and a refresh token, both stored in \texttt{httpOnly} cookies to prevent JavaScript XSS theft. The stateless access-token design requires no cross-instance session synchronisation.

\subsubsection{ASR-SEC-03 -- Brute-Force Protection}

The \texttt{LoginAttempt} PostgreSQL table records per-IP failure counts. After 5 failures within 15 minutes, the IP is blocked for the remainder of the lockout window (\texttt{LOCKOUT\_TIME\_MS = 900\,000\,ms}). On successful login, the record is deleted. Lockout state survives API restarts because it is database-persisted.

\subsubsection{ASR-SEC-05 -- Role-Based Access Control}

Every protected API route passes through Auth Middleware, which verifies the JWT signature and extracts the \texttt{role} claim. Endpoint-specific guards (\texttt{requireStudent}, \texttt{requireTeacher}, \texttt{requireAdmin}) enforce RBAC before any business logic executes, guaranteeing 100\,\% coverage.

\subsubsection{ASR-SEC-06 -- HTTPS/TLS}

Caddy terminates TLS for all inbound connections. For local development, \texttt{tls internal} generates a self-signed certificate. For production, Caddy automatically obtains and renews a certificate via ACME (Let's Encrypt). All inter-service traffic within the Docker network uses plain HTTP on the trusted internal network.

\subsubsection{ASR-SEC-07 -- Encryption at Rest}

PostgreSQL data-at-rest encryption is enabled at the volume level. Passwords are additionally protected by bcrypt hashing in the application layer --- even if the database is compromised, passwords remain computationally infeasible to recover.

\subsubsection{ASR-SEC-08 -- Input Validation}

Every API endpoint is guarded by a Zod schema middleware layer. Requests with invalid or missing fields are rejected with HTTP 400 before reaching the controller, preventing malformed data from entering business logic or the database.

\subsubsection{ASR-SEC-09 -- SQL Injection Prevention}

Prisma ORM is the exclusive database access layer. Prisma compiles all queries to parameterized statements; no raw string interpolation of user input into SQL is permitted. This provides structural protection independent of application-layer input validation.

\subsubsection{ASR-SEC-10 \& ASR-SEC-11 -- Audit Logging and Retention}

The Audit Module writes a canonical wide-event log entry for every security-sensitive action. Entries are append-only JSON lines stored in daily files on the \texttt{server-logs} volume. \texttt{AUDIT\_RETENTION\_DAYS=365} drives a scheduled clean-up job that removes entries older than one year.

\subsubsection{ASR-SEC-14 -- File Upload Size Limits}

Uploaded materials are limited to 50\,MB and submissions to 20\,MB. Limits are enforced at two layers:
\begin{enumerate}[noitemsep]
  \item \textbf{Client-side}: The File Upload Component checks size before initiating the HTTP request.
  \item \textbf{Server-side}: The \texttt{fileSizeLimit} middleware (Multer) rejects oversized payloads with HTTP 413.
\end{enumerate}

\subsubsection{ASR-SEC-15 -- API Rate Limiting}

The \texttt{rateLimiter} middleware intercepts every request to \texttt{/api/*}. It uses a Redis \texttt{INCR}/\texttt{EXPIRE} sliding-window counter keyed by \texttt{<endpoint>/<client-IP>}. Requests exceeding \texttt{RATE\_LIMIT\_MAX} receive HTTP 429. Because Redis counters are shared across both API instances, the limit is accurate even under load balancing.
