\section{Introduction}
% ══════════════════════════════════════════════════════════════════

\subsection{Purpose}

This Software Architecture Document (SAD) describes the software architecture of the
\textbf{Learning Management System (LMS)} developed as part of the UIT SE357 course project.
It establishes the high-level structure of the system, documents key architectural decisions,
and demonstrates how the design addresses the Architecturally Significant Requirements (ASRs)
with Status~=~Done.

The document is grounded in the actual project codebase (\texttt{server/},
\texttt{client/}, \texttt{docker-compose.yml}, \texttt{Caddyfile}) and is the
authoritative reference for the current release.

\subsection{Scope}

This document covers:

\begin{itemize}[noitemsep]
  \item A \textbf{React SPA} browser front end (\texttt{client/src/})
  \item A \textbf{Node.js / Express REST API} in two identical HA instances (\texttt{server/src/})
  \item A \textbf{PostgreSQL~16} relational database via \textbf{Prisma ORM} (\texttt{server/prisma/schema.prisma})
  \item A \textbf{Redis~7} in-memory store (\texttt{server/src/middlewares/redis.ts})
  \item A \textbf{Caddy~2} reverse proxy / load balancer / TLS terminator (\texttt{Caddyfile})
  \item A \textbf{File Storage Docker volume} for uploaded materials and submissions
\end{itemize}

Unimplemented ASRs (Status~$\neq$~Done), business processes, and infrastructure outside the
repository are out of scope.

\section{System Overview}
% ══════════════════════════════════════════════════════════════════

\subsection{System Description}

The \textbf{LMS} is a web-based platform for managing educational courses, classes, learning
materials, assignments, and student submissions. Three distinct user roles ---
Student, Teacher, and Admin --- are enforced by RBAC throughout.

The system is deployed as Docker containers orchestrated via Docker Compose and exposed to
end-users through a Caddy reverse proxy that handles TLS termination, gzip/zstd compression,
and active/passive load balancing.

  Teacher & \texttt{TEACHER} & Create and manage courses/classes; upload materials (PDF, VIDEO, LINK, DOC); create and grade assignments; provide feedback \\

  Email Service & External & Receives notification delivery requests via SMTP/Nodemailer from the API \\

\subsubsection{ASR-AVAIL-01 --- Availability \& Health-Check Failover}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.92\linewidth]{c3_avail_01}
  \caption{C4 Level~3 -- Availability: Dual Backend + Caddy Health-Check (ASR-AVAIL-01)}
  \label{fig:c3-avail}
\end{figure}

The diagram shows how Caddy's \texttt{lb\_policy first} keeps \texttt{backend-primary}
active and promotes \texttt{backend-secondary} when \texttt{GET /api/health} fails.
Both instances share PostgreSQL and Redis, making failover stateless and transparent.

\subsubsection{ASR-SEC-03 --- Brute-Force Login Protection}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.92\linewidth]{c3_sec_03}
  \caption{C4 Level~3 -- IP Lockout Logic (ASR-SEC-03)}
  \label{fig:c3-sec03}
\end{figure}

The diagram shows the component interactions inside the API for the login lockout flow:
\texttt{AuthRouter} $\rightarrow$ \texttt{AuthController} $\rightarrow$ \texttt{AuthService}
$\rightarrow$ \texttt{LoginAttempt} (PostgreSQL). After \texttt{MAX\_LOGIN\_ATTEMPTS}
consecutive failures from the same IP, \texttt{AuthService} sets \texttt{blockedUntil} and
all subsequent requests are rejected before password comparison.

\subsubsection{ASR-SEC-11 --- Audit Log Retention}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.92\linewidth]{c3_sec_11}
  \caption{C4 Level~3 -- Audit Log Retention Components (ASR-SEC-11)}
  \label{fig:c3-sec11}
\end{figure}

The diagram shows the \texttt{AuditLogger} utility writing JSONL entries to daily files,
\texttt{AuditConfig} resolving \texttt{AUDIT\_LOG\_DIR} and \texttt{AUDIT\_RETENTION\_DAYS},
and the daily cleanup job removing files older than the retention threshold.
The \texttt{AuditController} (\texttt{GET /api/audit/logs}) is admin-only and reads from
the same file store.

\subsubsection{ASR-SEC-14 --- File Upload Size Enforcement}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.92\linewidth]{c3_sec_14}
  \caption{C4 Level~3 -- File Upload Pipeline (ASR-SEC-14)}
  \label{fig:c3-sec14}
\end{figure}

The dual-layer enforcement is visible: the \textbf{client-side} File Upload Component
validates MIME type and file size before any HTTP request; the \textbf{server-side}
\texttt{enforceMaxUploadSize} middleware checks \texttt{Content-Length} and returns
HTTP~413 before Multer parses the body.

\subsubsection{ASR-SEC-15 --- Redis Rate Limiting}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.92\linewidth]{c3_sec_15}
  \caption{C4 Level~3 -- Redis-Backed Rate Limiting (ASR-SEC-15)}
  \label{fig:c3-sec15}
\end{figure}

The diagram shows the \texttt{App} class registering \texttt{rateLimiter} for
\texttt{/api/*}, the middleware calling \texttt{redisClient.incr(endpoint+"/"+ip)} and
\texttt{expire}, and returning HTTP~429 when the counter exceeds the configured limit.
Because Redis is shared across both API instances, the counter is accurate under load
balancing.

\section{Architecture Decisions}
\label{sec:adr}
% ══════════════════════════════════════════════════════════════════

\begin{tabularx}{\textwidth}{L{1.8cm} L{5cm} L{1.8cm} L{5cm}}
  \toprule
  \textbf{ADR} & \textbf{Title} & \textbf{Status} & \textbf{Addressed ASRs} \\
  \midrule
  ADR-001 & Caddy as Reverse Proxy, LB \& TLS  & Accepted & ASR-PERF-01, AVAIL-01, SEC-06 \\
  ADR-002 & JWT Access + Refresh Token Auth     & Accepted & ASR-SEC-01, SEC-02 \\
  ADR-003 & Redis Rate Limiting \& Login Lockout & Accepted & ASR-SEC-03, SEC-15 \\
  ADR-004 & PostgreSQL + Prisma ORM             & Accepted & ASR-PERF-02, SEC-07, SEC-09 \\
  ADR-005 & RBAC with JWT Role Claims           & Accepted & ASR-SEC-05 \\
  ADR-006 & Secure File Storage Outside Web Root & Accepted & ASR-SEC-14 \\
  ADR-007 & Canonical Audit Logging, 1-Year Retention & Accepted & ASR-SEC-10, SEC-11 \\
  \bottomrule
\end{tabularx}

\subsection{ADR-001: Caddy as Entry Point}

\textbf{Context:} Need a single TLS-terminating entry point that routes between two stateless
API instances and serves the React SPA as static files, with minimal configuration overhead.

\textbf{Decision:} Caddy~2 is the sole inbound gateway. It provides \texttt{tls internal}
(dev) / ACME auto-cert (prod), \texttt{encode gzip zstd}, active/passive load balancing
(\texttt{lb\_policy first}), health-check every 10\,s, and static SPA serving from
\texttt{/srv}.

\textbf{Consequences:} One decision simultaneously satisfies ASR-PERF-01, ASR-AVAIL-01,
and ASR-SEC-06. Caddy is itself a single point of failure (Risk~R-01).

\subsection{ADR-002: JWT Authentication}

\textbf{Context:} Two stateless API instances cannot share in-memory session state.

\textbf{Decision:} Passwords hashed with \texttt{bcrypt} ($\geq$10 salt rounds).
Login issues a JWT access token (1\,h, \texttt{AUTH\_SECRET}) and a refresh token
(\texttt{AUTH\_REFRESH\_SECRET}), both in \texttt{httpOnly} cookies. Either instance
can verify tokens independently without shared session store.

\textbf{Consequences:} Satisfies ASR-SEC-01 and ASR-SEC-02. Access tokens cannot be
revoked before expiry (Risk~R-04).

\subsection{ADR-003: Redis Rate Limiting}

\textbf{Context:} Rate limits must be accurate across two API instances; lockout state must
survive API restarts.

\textbf{Decision:} Redis \texttt{INCR}/\texttt{EXPIRE} backs the \texttt{rateLimiter}
middleware for cross-instance accuracy. Login lockout is additionally persisted in
PostgreSQL's \texttt{LoginAttempt} table for durability.

\textbf{Consequences:} Satisfies ASR-SEC-03 and ASR-SEC-15. Redis failure causes the rate
limiter to fail open (Risk~R-03).

\subsection{ADR-004: PostgreSQL + Prisma}

\textbf{Context:} Need a relational store with type-safe, injection-free queries and explicit
performance tuning.

\textbf{Decision:} PostgreSQL~16 is the sole relational store. Prisma ORM compiles all
queries to parameterized prepared statements. Explicit \texttt{@@index} directives on
high-cardinality FK columns support sub-2-second queries on 10\,000+ records.

\textbf{Consequences:} Satisfies ASR-PERF-02, ASR-SEC-07, ASR-SEC-09.

\subsection{ADR-005: RBAC}

\textbf{Context:} Three distinct roles (\texttt{STUDENT}, \texttt{TEACHER}, \texttt{ADMIN})
with non-overlapping permissions must be enforced uniformly before any business logic.

\textbf{Decision:} Role stored in the \texttt{User} model and in the JWT payload.
\texttt{AuthMiddleware.requireAdmin/Teacher/Student()} performs a live DB lookup before
every protected route's controller runs.

\textbf{Consequences:} Satisfies ASR-SEC-05. Each role-guard call adds one DB round-trip;
acceptable given \texttt{User} primary-key lookup latency.

\subsection{ADR-006: Secure File Storage}

\textbf{Context:} Uploaded files must not be directly accessible over HTTP and must respect
defined size limits.

\textbf{Decision:} Files stored in \texttt{/app/uploads} on the \texttt{server-logs}
Docker volume, outside Caddy's \texttt{/srv} web root. Size enforced at client
(File Upload Component) and server (\texttt{enforceMaxUploadSize} middleware).

\textbf{Consequences:} Satisfies ASR-SEC-14. Full malware scanning (ASR-SEC-12) and
guaranteed out-of-web-root enforcement (ASR-SEC-13) not yet complete (Risks~R-06, R-07).

\subsection{ADR-007: Canonical Audit Logging}

\textbf{Context:} Security-sensitive actions must be logged with structured metadata and
retained for at least one year.

\textbf{Decision:} \texttt{writeAuditLog()} appends JSON Lines to daily files on the
\texttt{server-logs} volume. \texttt{AUDIT\_RETENTION\_DAYS=365} drives an in-process
daily cleanup. Admin reads via \texttt{GET /api/audit/logs}.

\textbf{Consequences:} Satisfies ASR-SEC-10 and ASR-SEC-11. Logs are stored only locally
with no off-site backup (Risk~R-05).

\subsection{Security}

\subsubsection{ASR-SEC-01 \& SEC-02 --- Authentication}
Passwords hashed with \texttt{bcrypt} ($\geq$10 salt rounds) in \texttt{util/hash.ts}.
Login issues a JWT access token (1\,h) and refresh token in \texttt{httpOnly} cookies,
preventing JavaScript access.  Stateless design requires no cross-instance session sync.

\subsubsection{ASR-SEC-03 --- Brute-Force Protection}
\texttt{LoginAttempt} records per-IP failure counts.  After \texttt{MAX\_LOGIN\_ATTEMPTS}
failures, \texttt{blockedUntil = now() + LOCKOUT\_TIME\_MS} is set.  Lockout survives
API restarts because it is persisted in PostgreSQL.  Record deleted on successful login.

\subsubsection{ASR-SEC-05 --- RBAC}
Every protected route passes through \texttt{authenticateUser} then an appropriate role
guard (\texttt{requireAdmin/Teacher/Student}).  Guards perform a live DB lookup rather than
trusting only the JWT claim.  \texttt{requireAdmin} additionally emits an audit event on
every check, providing defence-in-depth.

\subsubsection{ASR-SEC-06 --- HTTPS/TLS}
Caddy terminates TLS for all inbound connections.  \texttt{tls internal} for local dev;
ACME (Let's Encrypt) for production.  Internal Docker network uses plain HTTP on the
trusted bridge.

\subsubsection{ASR-SEC-07 --- Encryption at Rest}
PostgreSQL data-at-rest encryption at the volume/OS level.  Passwords are additionally
bcrypt-hashed in the application layer.

\subsubsection{ASR-SEC-08 --- Input Validation}
\texttt{validateBody(schema)} (Zod) guards every API route.  HTTP~400 with structured
errors returned on invalid/missing fields before any business logic or DB call.

\subsubsection{ASR-SEC-09 --- SQL Injection Prevention}
Prisma ORM is the exclusive DB access layer.  All \texttt{findMany}/\texttt{create}/
\texttt{update}/\texttt{delete} calls compile to parameterized prepared statements.
No \texttt{\$queryRaw} with string interpolation exists in the codebase.

\subsubsection{ASR-SEC-10 \& SEC-11 --- Audit Logging and Retention}
\texttt{writeAuditLog()} called from controllers and middleware after every
security-sensitive action.  Entries are append-only JSONL in daily files.
\texttt{AUDIT\_RETENTION\_DAYS=365} drives \texttt{cleanupOldFiles()} which runs once
per calendar day, deleting expired files.

\subsubsection{ASR-SEC-14 --- File Upload Size Limits}
Enforced at two independent layers:
\begin{enumerate}[noitemsep]
  \item \textbf{Client}: File Upload Component checks \texttt{file.size} before the HTTP request
  \item \textbf{Server}: \texttt{enforceMaxUploadSize(maxBytes, name)} checks \texttt{Content-Length}; HTTP~413 if exceeded
\end{enumerate}

\subsubsection{ASR-SEC-15 --- API Rate Limiting}
\texttt{rateLimiter(rule)} middleware calls \texttt{redisClient.incr(endpoint+"/"+ip)} and
\texttt{expire}.  Counter key scopes by endpoint and IP.  Shared Redis ensures accuracy
across both API instances.
